<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Config file -> sync engine | Config file is operator-authored but loaded at runtime; malformed cron expressions or job names could cause panics |
| System clock -> scheduler | NTP corrections or VM time sync could cause clock jumps; scheduler must detect and handle gracefully |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | sync_config_to_db | mitigate | Config is already validated by parse_and_validate before reaching sync engine; cron expressions re-parsed by croner (which validates syntax). Invalid cron in DB would only occur if schema is tampered directly. |
| T-02-02 | Denial of Service | check_clock_jump | mitigate | Limit catch-up fire enumeration to a maximum window (e.g., 24 hours) to prevent unbounded iteration after long hibernation. Log WARN if window exceeds threshold. |
| T-02-03 | Information Disclosure | config_json in DB | mitigate | sync_config_to_db serializes job config excluding SecretString env values. Only env key names (not values) appear in config_json. Verified by unit test. |
</threat_model>
