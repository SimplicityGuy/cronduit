2. Create `src/scheduler/run.rs`:

   a. `pub async fn run_job(pool: DbPool, job: DbJob, trigger: String, cancel: CancellationToken) -> RunResult`:
   This is the per-run task lifecycle per SCHED-04:
   ```
   1. Insert running row: insert_running_run(&pool, job.id, &trigger)
   2. Create log channel: log_pipeline::channel(DEFAULT_CHANNEL_CAPACITY)
   3. Spawn log writer task (see below)
   4. Dispatch to executor based on job.job_type:
      - "command": parse config_json to get command string, call execute_command(cmd, timeout, cancel, sender)
      - "script": parse config_json to get script body + shebang, call execute_script(body, shebang, timeout, cancel, sender)

   5. Close the sender (signals log writer to finish)
   6. Wait for log writer task to complete
   7. Finalize run: finalize_run(&pool, run_id, status_str, exit_code, error_message)
   8. Return RunResult { run_id, status: status_str }
   ```

   b. Log writer task (spawned inside run_job):
   ```rust
   async fn log_writer_task(pool: DbPool, run_id: i64, receiver: LogReceiver) {
       loop {
           let batch = receiver.drain_batch_async(DEFAULT_BATCH_SIZE).await;
           if batch.is_empty() && receiver.is_empty_and_closed() {
               break;
           }
           if !batch.is_empty() {
               let tuples: Vec<(String, String, String)> = batch.into_iter()
                   .map(|l| (l.stream, l.ts, l.line))
                   .collect();
               if let Err(e) = insert_log_batch(&pool, run_id, &tuples).await {
                   tracing::error!(target: "cronduit.log_writer", run_id, error = %e, "failed to insert log batch");
               }
           }
       }
   }
   ```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-09 | Tampering | config_json parsing | mitigate | config_json is written by sync_config_to_db from validated config. Direct DB tampering is out of scope (operator has full DB access). Parse errors in run_job return Error status with message, not panic. |
| T-02-10 | Denial of Service | concurrent runs | accept | No concurrency limit in v1 per SCHED-06. Each run is a separate tokio task. At homelab scale, unbounded concurrency is acceptable. Monitoring via tracing. |
| T-02-11 | Denial of Service | log writer contention | mitigate | SQLite write pool has busy_timeout=5000ms (Phase 1). Micro-batch inserts (64 lines) are fast transactions. Per-run writer tasks queue naturally on the single-writer pool. |
</threat_model>
