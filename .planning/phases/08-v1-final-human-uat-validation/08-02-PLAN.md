<objective>
Ship TWO docker-compose example files so quickstart users can choose between the
simple "works on Linux with a single socket mount" path and the hardened "defense-in-depth
+ macOS-compatible" path. Both files must let cronduit's bollard client reach the
Docker daemon from inside its container — closing blocker #2 (hello-world docker.sock
Connect failure) for default compose users and giving Docker Desktop macOS operators
a working alternative.

- `examples/docker-compose.yml` stays the simple quickstart. Adds a numeric `group_add`
  stanza with `DOCKER_GID` default fallback and augments the SECURITY comment block to
  explain how to derive `DOCKER_GID` on Linux (`stat -c %g /var/run/docker.sock`).

- **NEW:** `examples/docker-compose.secure.yml` ships `tecnativa/docker-socket-proxy`
  as a sidecar and points cronduit at `DOCKER_HOST=tcp://dockerproxy:2375`, dropping
  the host socket mount from the cronduit service entirely. This is the "threat model
  honored" variant — narrow allowlist, no direct socket exposure, works on macOS Docker
  Desktop where socket ownership inside the Linux VM has no host GID mapping.

Purpose: The default quickstart promise is "git clone && docker compose up && you see
jobs running in the dashboard." The current docker-compose.yml mounts the socket but
the hello-world docker job fails because the nonroot container user isn't in the host
docker group. Adding `group_add` closes the gap for Linux. For macOS Docker Desktop
(and for production operators who want a real allowlist), the socket-proxy sidecar is
the correct solution.

Output: Two compose files that a new operator can `docker compose -f examples/X up`
and get working Docker job execution. The dual-file layout is exercised in CI by
Plan 08-04's `compose-smoke` matrix axis.
</objective>

    docker compose ignores it and it's deprecated.

    ```yaml
    # examples/docker-compose.secure.yml -- Hardened Cronduit quickstart.
    #
    # ============================================================================
    # SECURITY: THIS IS THE "THREAT MODEL HONORED" VARIANT
    # ============================================================================
    # This compose file runs cronduit with NO direct access to /var/run/docker.sock.
    # Instead, a `tecnativa/docker-socket-proxy` sidecar mediates every Docker API
    # call through a narrow allowlist: cronduit can list/create/start/wait/remove
    # containers and pull images, but it CANNOT access the rest of the Docker API
    # (networks, volumes, exec into containers, daemon info, swarm, system prune).
    #
    # Use this file when:
    #   * You are running on macOS / Docker Desktop. Inside the Linux VM the socket
    #     is root-owned and host GID mapping is unreliable, so the `group_add`
    #     approach from `docker-compose.yml` cannot work. The socket-proxy sidecar
    #     mounts the socket in ITS OWN root-owned container and exposes an HTTP
    #     allowlist to cronduit over a private Docker network.
    #   * You want defense-in-depth. Even a compromised cronduit (code execution
    #     via a malicious config reload, a bug in the web UI, etc.) cannot pivot
    #     to the full Docker API and root the host. Compromise is bounded to the
    #     allowlisted verbs.
    #   * You are fronting Cronduit with a reverse proxy and want the HTTP listener
    #     on loopback only. Replace `ports: - "8080:8080"` below with `expose: - "8080"`
    #     and put your proxy in a container on the `cronduit-net` network.
    #
    # See THREAT_MODEL.md in the repo root for the full threat surface. See
    # `examples/docker-compose.yml` for the simpler single-container variant.
    #
    # Prerequisites: Docker with Compose v2+
    # Usage: docker compose -f examples/docker-compose.secure.yml up -d
    # Web UI: http://localhost:8080

    services:
      # docker-socket-proxy: the ONLY container that mounts the host docker.sock.
      # Exposes a narrow HTTP allowlist on port 2375 inside the cronduit-net
      # network. Cronduit reaches it via DOCKER_HOST=tcp://dockerproxy:2375.
      #
      # Allowlist (everything else is default-deny in docker-socket-proxy):
      #   CONTAINERS=1  -- GET /containers/*, container inspect/list (read side)
      #   IMAGES=1      -- GET/POST /images/* (needed for bollard to pull hello-world)
      #   POST=1        -- enables POST verb on allowed endpoints, required for
      #                    POST /containers/create, /start, /wait
      #   DELETE=1      -- enables DELETE verb on allowed endpoints, required for
      #                    DELETE /containers/{id} (bollard::remove_container, fired
      #                    for `delete = true` docker jobs like the quickstart's
      #                    hello-world). Per 08-RESEARCH.md Q1 + D-29, POST=1 does
      #                    NOT imply DELETE in docker-socket-proxy — each HTTP verb
      #                    is a separate flag.
      #
      # NOT granted: NETWORKS, VOLUMES, EXEC, INFO, SWARM, SYSTEM, BUILD, SERVICES,
      # TASKS, NODES, PLUGINS, AUTH, SECRETS, CONFIGS, DISTRIBUTION, SESSION. If
      # cronduit ever needs more (e.g. to inspect a named network for
      # container:<name> mode), the error is loud and the allowlist can be extended
      # here in one place.
      dockerproxy:
        image: tecnativa/docker-socket-proxy:latest
        environment:
          - CONTAINERS=1
          - IMAGES=1
          - POST=1
          # DELETE=1 is required (D-29, resolves 08-RESEARCH.md Q1) because bollard's
          # remove_container call issues HTTP DELETE /containers/{id}; POST=1 alone
          # does NOT imply DELETE in docker-socket-proxy. Without this line, the
          # hello-world quickstart job (`delete = true`) fails at container removal.
          - DELETE=1
        volumes:
          - /var/run/docker.sock:/var/run/docker.sock:ro
        networks:
          - cronduit-net
        restart: unless-stopped
        # No host port binding: dockerproxy is ONLY reachable from inside the
        # cronduit-net network. Never expose port 2375 to the host.

      cronduit:
        image: ghcr.io/simplicityguy/cronduit:latest
        depends_on:
          - dockerproxy
        ports:
          - "8080:8080"
        volumes:
          # No /var/run/docker.sock mount -- all Docker API traffic flows through
          # dockerproxy over the private network.
          - ./cronduit.toml:/etc/cronduit/config.toml:ro
          - cronduit-data:/data
        environment:
          - RUST_LOG=info,cronduit=debug
          - DATABASE_URL=sqlite:///data/cronduit.db
          # Point bollard at the socket-proxy over the private network. bollard
          # reads DOCKER_HOST the same way the docker CLI does.
          - DOCKER_HOST=tcp://dockerproxy:2375
        networks:
          - cronduit-net
        restart: unless-stopped

    networks:
      cronduit-net:
        driver: bridge

    volumes:
      cronduit-data:
    ```

    Notes:
    - `ports: - "8080:8080"` is preserved to match the quickstart promise in the
      default compose file (Phase 6 D-12 decision). An operator who wants loopback-only
      flips it to `expose:` per the SECURITY comment.
    - The `dockerproxy` service has NO host port binding — port 2375 is only on the
      internal `cronduit-net` network. The socket never exits the sidecar.
    - `depends_on: [dockerproxy]` is order-only; compose will NOT wait for health. The
      pre-flight Docker ping in Plan 08-03 handles the race at startup by retrying.
    - The `cronduit-data` volume name is shared with the default compose file so users
      switching between the two compose variants keep their data. This is intentional.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host -> cronduit container | Docker socket access (default compose: direct mount + group_add; secure compose: mediated through docker-socket-proxy) |
| cronduit container -> dockerproxy sidecar | HTTP API traffic over private Docker network on port 2375 |
| dockerproxy sidecar -> host Docker daemon | Read-only mount of /var/run/docker.sock |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-05 | Elevation of Privilege | cronduit has full Docker API access via mounted socket (default compose) | accept | Intentional quickstart trade-off, documented in augmented SECURITY block. Linux operators using group_add get host docker-group membership — effectively root-equivalent via `docker run --privileged --pid=host`. The SECURITY block explicitly points high-risk deployments at docker-compose.secure.yml. |
