<objective>
Implement SSE (Server-Sent Events) log streaming for in-progress runs on the Run Detail page.

Purpose: Operators need real-time log visibility for running jobs (UI-14). The existing static log viewer only shows completed runs. SSE provides live streaming without WebSocket complexity.

Output: A `GET /events/runs/:id/logs` SSE endpoint, a broadcast channel fan-out system integrated into the run lifecycle, a Run Detail page that conditionally renders live SSE or static logs based on run status, and SSE integration test stubs.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SSE client to server | Any HTTP client can open SSE connections to /events/runs/:id/logs |
| Log content to SSE stream | Raw log output (potentially attacker-controlled) flows through SSE to browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-6-01 | Information Disclosure | SSE endpoint | accept | SSE endpoint is read-only and requires valid run_id (integer path param). Returns immediate close for non-existent runs. No auth in v1 (documented stance per THREAT_MODEL.md). Consistent with all other endpoints. |
| T-6-02 | Denial of Service | SSE broadcast | mitigate | Broadcast channel capacity 256; slow subscribers get Lagged error not backpressure. KeepAlive detects dead connections. Broadcast sender dropped when run finalizes, closing all subscriber streams. |
| T-6-03 | Cross-Site Scripting | SSE log content | mitigate | All log content HTML-escaped via html_escape() in format_log_line_html() before insertion into SSE data field. No raw HTML or script injection possible through log lines. Uses textContent for error messages in JS. |
</threat_model>
