```rust
pub struct DashboardJobView {
    pub id: i64,
    pub name: String,
    pub schedule: String,
    pub resolved_schedule: String,
    pub next_fire: String,        // e.g., "in 4h 12m"
    pub last_status: String,      // e.g., "success", "failed", "running", ""
    pub last_status_label: String, // e.g., "SUCCESS", "FAILED", "RUNNING"
    pub last_run_relative: String, // e.g., "2m ago"
}
```

Create a function `to_view(job: DashboardJob, tz: chrono_tz::Tz) -> DashboardJobView` that:
- Computes `next_fire` using `croner::Cron::new(&job.resolved_schedule).find_next_occurrence(&now_tz)` and formats as relative time (e.g., "in 4h 12m", "in 30s", "in 2d 5h"). Use a helper function `format_relative_time(dt: DateTime, now: DateTime) -> String`.
- Computes `last_status` from the DB field (lowercase for CSS class matching: "success", "failed", "running", "timeout", "error").
- Computes `last_status_label` as uppercase: "SUCCESS", "FAILED", etc.
- Computes `last_run_relative` from `last_run_time` as relative time ago (e.g., "2m ago", "3h ago", "1d ago"). If None, returns "never".

The handler function:
```rust
pub async fn dashboard(
    HxRequest(is_htmx): HxRequest,
    State(state): State<AppState>,
    Query(params): Query<DashboardParams>,
) -> impl IntoResponse {
    let filter = if params.filter.is_empty() { None } else { Some(params.filter.as_str()) };
    let jobs = queries::get_dashboard_jobs(&state.pool, filter, &params.sort, &params.order)
        .await
        .unwrap_or_default();

    // Get timezone from config — for now use UTC as default
    // TODO: pass tz through AppState in a future refinement
    let tz: chrono_tz::Tz = chrono_tz::UTC;
    let job_views: Vec<DashboardJobView> = jobs.into_iter().map(|j| to_view(j, tz)).collect();

    let csrf_token = crate::web::csrf::get_or_create_token(&state);

    if is_htmx {
        JobTablePartial { jobs: job_views, csrf_token }.into_response()
    } else {
        DashboardPage {
            jobs: job_views,
            filter: params.filter,
            sort: params.sort,
            order: params.order,
            config_path: state.config_path.display().to_string(),
            csrf_token,
        }.into_response()
    }
}
```

Note: The CSRF module is created in Plan 05. For now, use a placeholder that generates a random hex string. The dashboard handler can use `hex::encode(rand::random::<[u8; 16]>())` as a temporary token until the full CSRF middleware is wired.

| browser -> GET / | Dashboard page render; read-only data from DB |
| browser -> GET /partials/job-table | HTMX partial; same data, no state changes |
| filter input -> SQL query | User-provided filter string used in LIKE query |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-08 | Injection | filter param in dashboard query | mitigate | Parameterized LIKE query from Plan 02; filter value bound as `?` parameter, never interpolated. |
| T-03-09 | Injection | sort/order params | mitigate | Whitelist match in Plan 02 query function; user input never placed in SQL string. |
| T-03-10 | Spoofing | HTMX polling source | accept | No auth in v1; polling is read-only. Documented in THREAT_MODEL.md. |
