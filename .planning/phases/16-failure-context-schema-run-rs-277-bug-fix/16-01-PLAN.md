**Task count rationale (8 tasks):** This plan is the schema-substrate convergence point — it lands 6 mechanical migration files (one DDL or one bulk-UPDATE statement each, paired across two backends), 1 integration test, and 1 gate-verification step. Splitting T1-T6 into multiple plans would break the parity-verification invariant mid-plan (the SQLite and Postgres halves of each migration file pair must land together to keep `tests/schema_parity.rs` green at every commit boundary). T7 (the test) and T8 (the green-bar gate) intentionally trail the migrations so the verification only fires once the structural substrate is fully in place. Each task is small (a single SQL statement or one test scaffold) and the plan as a whole stays well under the 50% context-window target despite the count.

<objective>
Land the FCTX-04 schema substrate: add `job_runs.image_digest TEXT NULL` (FOUND-14 storage) and `job_runs.config_hash TEXT NULL` (FCTX-04 storage) on both SQLite and Postgres, then best-effort backfill `config_hash` from the existing per-JOB `jobs.config_hash` for pre-v1.2 rows. All three migrations land per backend (6 files total), with the backfill carrying the BACKFILL_CUTOFF_RFC3339 structured-comment marker (D-03) that Phase 21's UI panel will read to distinguish backfilled rows from authentic per-run captures.

Purpose: Without these columns existing on both backends, Plan 16-04's `finalize_run` and `insert_running_run` signature changes have nowhere to write. Without the backfill marker, Phase 21 cannot reliably distinguish authentic per-run captures from "config_hash at backfill time" approximations, causing misleading "config changed: No" deltas across the upgrade boundary.

Output: 6 NEW migration files + 1 NEW integration test file. Zero edits to `tests/schema_parity.rs` (RESEARCH §E — dynamic introspection already accepts new TEXT columns automatically).
</objective>
