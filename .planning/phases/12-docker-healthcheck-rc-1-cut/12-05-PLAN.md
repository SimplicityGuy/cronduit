From RESEARCH.md §Pattern 4 (the exact patch shape):
```yaml
          tags: |
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}},enable=${{ !contains(github.ref, '-') }}
            type=semver,pattern={{major}},enable=${{ !contains(github.ref, '-') }}
            type=raw,value=latest,enable=${{ !contains(github.ref, '-') }}
            type=raw,value=rc,enable=${{ contains(github.ref, '-rc.') }}
```

Also note: the existing prerelease routing (line 163, approx) is `prerelease: ${{ contains(steps.version.outputs.version, '-') }}` and is ALREADY correct for rc tags — Plan 05 does NOT touch it.

Why `!contains(github.ref, '-')`: GitHub refs for tags are `refs/tags/v1.1.0` (release) or `refs/tags/v1.1.0-rc.1` (pre-release). The hyphen only appears in pre-release semver. Per RESEARCH.md, this is cleaner than a regex.

Why `contains(github.ref, '-rc.')` (with the dot): catches `v1.1.0-rc.1` but NOT a hypothetical `v1.1.0-beta.1`. Phase 12 only ships rc tags. If Phase 13/14 ever introduce beta tags they'd need their own raw line.

RESEARCH.md note: docker/metadata-action's `{{major}}` and `{{major}}.{{minor}}` patterns ALREADY auto-skip pre-releases. The `enable=` clauses on those two lines are belt-and-suspenders. CONTEXT.md D-10 explicitly requests them anyway for documentation clarity, so keep them.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Git tag (`refs/tags/vX.Y.Z[-rc.N]`) → metadata-action expressions → GHCR tag set | The git tag is the trust anchor; D-13 (Plan 07) keeps tag creation maintainer-side, NOT workflow-dispatch. |
| `github.ref` value → `enable=` expression evaluation → tag inclusion/exclusion | `github.ref` is server-controlled (not user-controlled); standard GHA workflow injection guidance applies (see RESEARCH.md §Security Domain). |
| Pushed image tags → `:latest` consumers (existing operators on v1.0.1) | The `:latest` invariant is the operator-trust contract: `:latest` MUST stay at v1.0.1 until final v1.1.0. The patch enforces it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-05-01 | Tampering | `:latest` drift to a pre-release | mitigate | The `enable=${{ !contains(github.ref, '-') }}` gate on `type=raw,value=latest` blocks any tag with a hyphen (which is every pre-release semver). Acceptance criterion verifies the literal expression is present. |
| T-12-05-02 | Tampering | Workflow injection via `github.ref` | accept | `github.ref` is server-controlled metadata (matches `refs/tags/*` for tag pushes); not attacker-controllable. Standard GHA pattern; RESEARCH.md §Security Domain confirms safe. |
| T-12-05-03 | Information Disclosure | Workflow expression syntax errors silently produce no tags | mitigate | YAML-parse acceptance criterion catches syntax errors locally; metadata-action documented behavior is to fail-loud on a broken `enable=` expression. The first rc.1 tag push (Plan 07 UAT) is the runtime validation. |
| T-12-05-04 | Repudiation | Untraceable tag-template intent | mitigate | The new comment block above `tags:` enumerates the per-template behavior in two scenarios (final release + pre-release) so a future reader (or auditor) can reconstruct the intent without re-running the action. |
| T-12-05-05 | Elevation of Privilege | rc tag triggers `:latest` push | mitigate | Same gate as T-12-05-01; the regression-locked test is the GHCR manifest inspection in the Plan 07 runbook (`docker manifest inspect ghcr.io/.../cronduit:latest` digest must equal the v1.0.1 digest after the rc.1 push). |
