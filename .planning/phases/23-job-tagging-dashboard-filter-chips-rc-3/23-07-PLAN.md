## Scenario 1 — Chip strip render + alphabetical + empty-state hidden (TAG-06 / D-17 step 1)

**Goal:** Confirm the chip strip renders one chip per distinct fleet tag, alphabetical, and is hidden entirely when no job has tags.

**Steps:**

1. Run `just uat-chips-render` from a fresh terminal.
2. Follow the recipe's prompts (build → db-reset → write fleet TOML → run cronduit in another terminal → open dashboard).

**Sign-off:**

- [ ] Scenario 1 passed (chip strip renders alphabetically; empty-state hides the strip).

## Scenario 2 — AND-filter + untagged-hidden + name-filter composition (TAG-06 + TAG-07 / D-17 step 2)

**Goal:** Confirm multi-chip AND semantics + untagged-hidden when filter active + AND with v1.0 name-filter.

**Steps:**

1. Run `just uat-chips-and-filter`.
2. Follow the recipe through (a) clicking `backup` → 3 rows visible, untagged hidden; (b) also clicking `weekly` → 2 rows visible (AND); (c) typing `prod` in name-filter → 1 row visible (AND with name); (d) deactivating `weekly` → 2 rows visible.

**Eyeball criteria:**

- **(a)** clicking `backup` → chip turns teal-bordered + bold (UI-SPEC § Color active state). Three rows visible: `prod-backup-weekly`, `prod-backup-only`, `dev-backup-weekly`. The `untagged-noise` row is HIDDEN.
- **(b)** clicking `weekly` too → both teal + bold. TWO rows: `prod-backup-weekly`, `dev-backup-weekly`.
- **(c)** typing `prod` in name-filter → ONE row: `prod-backup-weekly`.
- **(d)** clicking `weekly` again → returns to grey. Now `prod-backup-only` reappears (matches `prod` name + `backup` tag).

**Sign-off:**

- [ ] Scenario 2 passed (AND across chips, AND with name-filter, untagged-hidden when filter active).

**Sign-off:**

- [ ] Scenario 3 passed (shareable URL round-trips; stale tags silently dropped).

**Eyeball criteria:**

- Chip strip uses `flex-wrap` to multiple rows — every chip remains visible (D-03).
- NO horizontal scroll on the chip strip.

**Sign-off:**

- [ ] Scenario 4 passed (chip strip reflows cleanly on mobile; touch targets remain ≥ 44px).

## Scenario 5 — Light-mode parity (TAG-06 + UI-SPEC § Color / D-17 light mode)

**Goal:** Confirm chip rendering honors `[data-theme="light"]` automatically (no new light-mode work).

**Steps:**

1. With cronduit running and a multi-tag fleet loaded, ensure the dashboard is in dark mode (default).
2. Toggle to light mode via the existing theme-switch primitive (whatever button or system-setting integration the v1.0/v1.1 dashboard uses; verify by inspecting `<html data-theme="...">` in DevTools).
3. Reload the dashboard.

**Eyeball criteria:**

- Hover/focus states behave consistently with light-mode tokens.
- Three-channel active encoding (border + label color + bold weight) remains visible in light mode.

**Sign-off:**

- [ ] Scenario 5 passed (light mode renders chip strip with correct token mapping; no hardcoded dark-mode colors).

## Scenario 6 — Keyboard navigation + screen-reader narration (TAG-06 + UI-SPEC § Accessibility / D-17 keyboard + screen-reader)

**Goal:** Confirm WCAG 2.2 AAA touch + keyboard + AT contracts.

**Steps:**

1. With cronduit running and a multi-tag fleet loaded, focus the browser tab.
2. Press Tab repeatedly to navigate from the URL bar onto the dashboard's interactive elements; confirm the chip anchors are reachable in document order.
3. Tab onto a chip; press Enter (and separately, press Space). Both MUST trigger the chip toggle (URL changes; chip active state updates via OOB swap).
4. Activate a screen reader (VoiceOver on macOS, NVDA on Windows, Orca on Linux). Navigate to the chip strip; listen.

**Eyeball / ear criteria:**

- **(a)** Tab order: chip strip is reachable BEFORE the name-filter input (matches D-01 visual order).
- **(b)** `:focus-visible` ring renders on every focused chip (`box-shadow: 0 0 0 2px var(--cd-green-dim)` per UI-SPEC § Color).

- **(d)** Screen reader announces the strip group label: "Filter jobs by tag, group" (or platform-equivalent for `role="group" aria-label="Filter jobs by tag"`).
- **(e)** Screen reader announces each chip with the full aria-label: "Tag filter: backup (inactive — click to add)" or "Tag filter: backup (active — click to remove)" depending on state.
- **(f)** Screen reader announces the `aria-pressed` state ("pressed" / "not pressed" or platform-equivalent).
- **(g)** No focus traps — Tab from the last chip moves into the name-filter input.
- **(h)** Touch target — focus the chip and inspect the bounding box in DevTools; confirm the rendered height is ≥ 44px (chip's effective tap surface).

**Sign-off:**

- [ ] Scenario 6 passed (keyboard navigation, screen-reader narration, touch targets, focus rings all conform to UI-SPEC § Accessibility Contract).
