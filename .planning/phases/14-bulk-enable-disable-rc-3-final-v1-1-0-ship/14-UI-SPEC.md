# Phase 14 — UI Design Contract (Bulk Enable/Disable, rc.3 + Final v1.1.0 Ship)

> Visual and interaction contract for the four UI surfaces delivered in Phase 14: (1) dashboard row-checkbox column + select-all, (2) sticky inline bulk-action bar, (3) toast copy for the bulk-toggle API responses, (4) settings "Currently Overridden" audit section. Generated by gsd-ui-researcher, verified by gsd-ui-checker. All decisions below are (a) locked upstream in `14-CONTEXT.md` (D-01..D-21), (b) carried forward from Cronduit's shipped `design/DESIGN_SYSTEM.md` + live `assets/src/app.css` tokens, or (c) resolved by this spec for the explicit "Claude's Discretion" sub-areas called out in `14-CONTEXT.md` § `<decisions>`.

**Upstream authority:** Phase 14 CONTEXT.md locks 21 design decisions (D-01..D-21). This UI-SPEC translates those into exact tokens, exact copywriting, exact selectors, and interaction state tables. It does **not** re-open any locked decision.

**Phase 13 parity:** This spec follows the same shape, rigor, and additive-only philosophy as `.planning/phases/13-observability-polish-rc-2/13-UI-SPEC.md`. All existing `cd-badge`, `cd-btn-*`, `cd-status-*`, `cd-sparkline-*`, `cd-timeline-*`, `cd-pill-*` selectors and tokens are untouched.

## Typography

All new surfaces inherit JetBrains Mono from `base.html`. Phase 14 uses exactly **4 distinct size tokens**; both active weights (400 regular + 700 bold) are already loaded. The 4-size scale is: `xs` (label), `sm` (body), `base` (primary body), `lg` (section heading).

| Role | Token | Size | Weight | Line Height | Letter Spacing | Phase 14 usage |
|------|-------|------|--------|-------------|----------------|----------------|
| Label / column header / badge | `--cd-text-xs` | 10.4px (0.65rem) | 700 | 1.5 | `0.1em` UPPERCASE | Settings table column headers `NAME` / `OVERRIDE STATE` / `CLEAR`; DISABLED / FORCED ON badge text; settings section description-line label treatment |
| Secondary / button / description | `--cd-text-sm` | 12.8px (0.8rem) | 400 | 1.5 | `0` | Bulk-bar button labels (`Disable selected`, `Enable selected`, `Clear`); settings section description line (`These jobs have an override set…`); per-row `Clear` button |
| Body / selection count | `--cd-text-base` | 14.4px (0.9rem) | 400 | 1.6 | `0` | Bulk-bar selection count (`{N} selected`); settings table body cells (job name column content); per-row name link text (matches `partials/job_table.html:4` job-name pattern) |
| Section heading | `--cd-text-lg` | 20px (1.25rem) | 700 | 1.4 | `-0.02em` | Settings section `<h2>Currently Overridden</h2>` (CONTEXT D-09 specifies `--cd-text-lg` explicitly) |

**Count:** exactly 4 distinct size tokens (`xs`, `sm`, `base`, `lg`). `--cd-text-xl` and `--cd-text-md` are NOT used anywhere in Phase 14; executor MUST NOT introduce additional size tokens. 2 weights in active use (`400` regular + `700` bold); weight 500 medium is NOT loaded by `@font-face` and MUST NOT be specified.

**Why `--cd-text-lg` here (vs. Phase 13's `--cd-text-xl`):** CONTEXT D-09 explicitly dictates `font-size:var(--cd-text-lg);font-weight:700;margin-top:var(--cd-space-8)` for the settings section heading. Phase 13 used `--cd-text-xl` for BOTH page `<h1>` and card `<h2>` because each of its three surfaces was effectively a new top-level surface; Phase 14's new section is clearly subordinate to the existing `<h1>Settings` and must visually reflect that.

**Dashboard surfaces (bulk bar + row checkboxes):** No heading introduced. The bar is chrome; labels are `xs` + `sm` + `base`. No new typography hierarchy on the dashboard.

### Accent reserved-for list (explicit — 10% accent discipline)

`--cd-text-accent` in Phase 14 is used for EXACTLY these elements:

1. **Checkbox native theming** via `accent-color: var(--cd-text-accent)` on `.cd-row-checkbox` and `#cd-select-all` — ticked checkboxes fill with brand green on every modern browser (Chromium 93+, Firefox 92+, Safari 15.4+).
2. **Settings overridden-list job-name link** (`<a class="text-(--cd-text-accent) hover:underline font-bold">{{ job.name }}</a>`) — mirrors the shipped `partials/job_table.html:4` convention verbatim.
3. **`cd-btn-primary` background** — only used transitively if the planner chooses to render the "Disable selected" button as primary (see Claude's-Discretion resolution below — recommended: `cd-btn-secondary` for both Enable + Disable to avoid destructive-primary pattern; Clear as `cd-btn-secondary` always).

Accent is NOT used for: bulk-bar surface; action-bar button backgrounds (both default to `cd-btn-secondary`); selection-count label text; DISABLED / FORCED ON badge text (they use their own status colors); table body text; row hover.

### Destructive token usage

CONTEXT D-07 locks bulk-disable as **idempotent, not confirmation-gated**, consistent with Run Now and Stop. No destructive-confirmation modal is introduced. However, "Disable selected" IS semantically destructive (stops a job firing); the 60/30/10 rule requires that destructive actions use a consistent pattern. Phase 14 adopts this contract:

- **"Disable selected" button:** `cd-btn-secondary` base, with `:hover` tinting toward `--cd-status-disabled-bg` + `--cd-status-disabled` text color (mirrors the shipped `cd-btn-stop:hover` pattern at `app.css:258-262` which tints toward `--cd-status-stopped-bg`). Locked selector: `.cd-btn-secondary.cd-btn-disable-hint` (NEW additive modifier). Rationale: gives a visual "are you sure about this" lean on hover without adding a modal.
- **"Enable selected" button:** `cd-btn-secondary` unchanged (no hover tint). Enable is a no-op-to-positive-effect action; default chrome is correct.
- **"Clear" (selection) button:** `cd-btn-secondary` unchanged. Clear affects only UI selection state, not the DB.
- **Per-row "Clear" button in settings:** `cd-btn-secondary text-sm py-1 px-3` (matches `partials/run_history.html:79` pattern for existing secondary actions).

Alternative considered: make "Disable selected" the `cd-btn-stop` variant (which already has a hover-tint pattern toward `--cd-status-stopped-bg`). Rejected because `cd-btn-stop` visually maps to the Phase 10 Stop button (kills a running job) and reusing it here would create a "Stop == Disable" visual confusion. A new additive `.cd-btn-disable-hint` modifier keeps the semantic clear.

### Toast copy (HX-Trigger showToast)

Honors D-04 + D-06 + D-12 verbatim. Existing toast system at `base.html:66-118` already routes `level: 'info'` vs `level: 'error'` to the correct color variant; Phase 14 only specifies the `message` string + the `level` value.

| Scenario | `message` | `level` | `duration` |
|----------|-----------|---------|------------|
| Bulk disable, N ticked, M running (M == 0) | `{N} jobs disabled.` | `info` | default `3000` |
| Bulk disable, N ticked, M running (M > 0, M == 1) | `{N} jobs disabled. 1 currently-running job will complete naturally.` | `info` | default `3000` |
| Bulk disable, N ticked, M running (M > 0, M > 1) | `{N} jobs disabled. {M} currently-running jobs will complete naturally.` | `info` | default `3000` |
| Bulk disable, N ticked, K invalid (K > 0) | `{N - K} jobs disabled. ({K} not found)` — with the running-job clause prefixed BEFORE the `({K} not found)` suffix when M > 0, e.g. `2 jobs disabled. 1 currently-running job will complete naturally. (1 not found)` | `info` | default `3000` |
| Bulk disable, N == 1, singular | `1 job disabled.` (singular `job`, not `jobs`) | `info` | default `3000` |
| Bulk enable, N ticked | `{N} jobs: override cleared.` | `info` | default `3000` |
| Bulk enable, N == 1 | `1 job: override cleared.` (singular) | `info` | default `3000` |
| Bulk enable, K invalid | `{N - K} jobs: override cleared. ({K} not found)` | `info` | default `3000` |
| Per-row Clear (from settings, N always == 1) | `Override cleared.` (no count — single-row context is unambiguous) | `info` | default `3000` |
| CSRF mismatch | `Session expired — refresh and try again.` | `error` | `0` (sticky — error toast per `base.html:95-107` stays until dismissed) |
| Invalid `action` (neither `disable` nor `enable`) | `Invalid bulk action.` | `error` | `0` |
| Empty `job_ids` (operator managed to POST zero) | `No jobs selected.` | `error` | `0` (see Claude's-Discretion resolution below — locks 400 + error toast over silent 200) |
| Scheduler unavailable (mpsc closed, 503) | `Scheduler is shutting down — try again shortly.` | `error` | `0` |

**Singular/plural rule:** `N == 1 && K == 0` → `job` (singular); otherwise → `jobs` (plural). `M == 1` → `currently-running job will complete naturally`; otherwise → `currently-running jobs will complete naturally`. Implemented via askama conditional in the toast-trigger payload (executor's call whether to branch in the handler or use a helper).

**Inline JS helpers** (NEW additive script; lives in a small `<script>` block at the bottom of `templates/pages/dashboard.html`, BEFORE `{% endblock %}`):
```html
<script>
  function __cdBulkSelectAll(headerCheckbox) {
    var rows = document.querySelectorAll('.cd-row-checkbox:not(#cd-select-all)');
    rows.forEach(function(cb) { cb.checked = headerCheckbox.checked; });
    __cdBulkUpdateBar();
  }
  function __cdBulkOnRowChange() {
    __cdBulkUpdateIndeterminate();
    __cdBulkUpdateBar();
  }
  function __cdBulkUpdateIndeterminate() {
    var header = document.getElementById('cd-select-all');
    if (!header) return;
    var rows = document.querySelectorAll('.cd-row-checkbox:not(#cd-select-all)');
    var checked = 0;
    rows.forEach(function(cb) { if (cb.checked) checked++; });
    header.indeterminate = (checked > 0 && checked < rows.length);
    header.checked = (checked > 0 && checked === rows.length);
  }
  function __cdBulkUpdateBar() {
    var bar = document.getElementById('cd-bulk-action-bar');
    if (!bar) return;
    var rows = document.querySelectorAll('.cd-row-checkbox:not(#cd-select-all):checked');
    var count = rows.length;
    var countEl = document.getElementById('cd-bulk-count');
    if (countEl) countEl.textContent = String(count);
    bar.hidden = (count === 0);
  }
  // After every HTMX swap of the table body (3s poll, filter change), re-sync the header checkbox
  // indeterminate state in case rows are added/removed under us.
  document.body.addEventListener('htmx:afterSwap', function(e) {
    if (e.target && e.target.id === 'job-table-body') { __cdBulkUpdateIndeterminate(); __cdBulkUpdateBar(); }
  });
</script>
```

**Why inline JS is acceptable here:** CLAUDE.md locks "no new JS beyond vendored HTMX". This script is ~30 lines of vanilla DOM manipulation (no new libraries, no new dependencies, no frameworks). The equivalent HTMX-native approach would require custom HTMX extensions that increase complexity. This is the same category of thin inline script Phase 11 used for the log-dedupe client (`templates/pages/run_detail.html` data-max-id handler). Resolves CONTEXT Claude's-Discretion "Indeterminate-state UI for the header checkbox" in favor of explicit state sync rather than CSS-only (which can't set `indeterminate`).

### Surface C: Toast behavior (ERG-02, D-04, D-12b)

**Mechanism:** The existing toast system in `templates/base.html:66-118` listens for the `showToast` custom event and renders a toast element from `{detail: {message, level, duration}}`. Phase 14's handler fires this via an HTMX `HX-Trigger` response header containing the event payload as JSON — identical to the shipped `run_now` / `stop_run` / `reroll` handlers.

**HX-Trigger payload shape** (set on the `bulk_toggle` 200 response):
```
HX-Trigger: {"showToast":{"message":"3 jobs disabled. 1 currently-running job will complete naturally.","level":"info","duration":3000}}
```

For errors (CSRF mismatch, 503, empty `job_ids`):
```
HX-Trigger: {"showToast":{"message":"Session expired — refresh and try again.","level":"error","duration":0}}
```

**Response shape:** Per CONTEXT D-12b, the handler returns **empty 200** (no body) with only the `HX-Trigger` header. The existing 3s HTMX table-body poll picks up any DB-side changes (override state reflected in the sparkline / badge rendering if the executor chooses to surface override-state on the dashboard — not required by ERG-01 but a discretionary enhancement). No `HX-Refresh: true` (CONTEXT explicitly rejects this as "jarring mid-interaction").

**Toast visual contract (already in `base.html:66-118`; Phase 14 changes nothing):**

| Level | Background | Text | Border | Duration | Dismiss |
|-------|-----------|------|--------|----------|---------|
| `info` | `--cd-status-active-bg` | `--cd-status-active` | `1px solid --cd-status-active` | 3000ms auto-dismiss | Fade out 300ms |
| `error` | `--cd-status-error-bg` | `--cd-status-error` | `1px solid --cd-status-error` | `0` sticky | `×` close button |

**Toast position:** `fixed; top: 1rem; right: 1rem; z-index: 50` — set in `base.html:59` on `<div id="toast-container">`. Phase 14 unchanged.
