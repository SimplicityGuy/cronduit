Create `src/scheduler/random.rs` with these public functions:

1. `pub fn is_random_schedule(schedule: &str) -> bool` -- returns true if any whitespace-delimited field equals `@random`.

2. `pub fn resolve_schedule(raw: &str, existing_resolved: Option<&str>, rng: &mut impl Rng) -> String` -- for each field in the 5-field cron string, if the field is `@random`, pick a random value from the valid range for that position (minute: 0-59, hour: 0-23, dom: 1-31, month: 1-12, dow: 0-6). If `existing_resolved` is `Some` AND the raw schedule hasn't changed (caller determines this via config_hash), return the existing value. For non-`@random` fields, pass through unchanged. After resolution, validate the result with `croner::Cron::from_str()` and check `find_next_occurrence()` returns within 366 days. If invalid, retry with new random values (up to 10 attempts), then accept best candidate.

   Use these field ranges (constant array):
   ```rust
   const FIELD_RANGES: [(u32, u32); 5] = [
       (0, 59),   // minute
       (0, 23),   // hour
       (1, 31),   // day of month
       (1, 12),   // month
       (0, 6),    // day of week
   ];
   ```

3. `pub fn resolve_random_schedules_batch(jobs: &[(String, String, Option<String>)], min_gap: Duration, rng: &mut impl Rng) -> Vec<(String, String)>` -- takes (job_name, raw_schedule, existing_resolved), returns (job_name, resolved_schedule). Implements slot-based gap enforcement:

   a. Separate jobs into random and non-random. Non-random jobs get identity resolution.
   b. Sort random jobs by constraint severity (fewer @random fields = more constrained = allocate first).
   c. Feasibility pre-check: if `num_random_jobs * min_gap_minutes > available_minutes_in_day (1440)`, log `tracing::warn!(target: "cronduit.random", jobs = num_random_jobs, gap_minutes = min_gap.as_secs() / 60, "random_min_gap is infeasible; relaxing gap for overflow jobs")` and compute a relaxed gap = 1440 / num_random_jobs minutes.
   d. For each random job, call resolve_schedule to get a candidate. Extract the effective fire minute-of-day (hour * 60 + minute). Check against already-allocated slots. If too close, retry up to 100 times. If all fail, accept the candidate with maximum minimum distance to neighbors and log WARN.
   e. Record the allocated slot for gap checking.

4. Add `pub mod random;` to `src/scheduler/mod.rs` module declarations.

Import `rand::Rng`, `std::time::Duration`, `tracing`. Use `rand::thread_rng()` in non-test code; tests use `rand::rngs::StdRng::seed_from_u64(42)` for determinism.

3. Before the per-job loop, build the batch resolver input:
   ```rust
   let mut rng = rand::thread_rng();
   // Build input for batch resolution: (name, raw_schedule, existing_resolved_from_db)
   let mut batch_input: Vec<(String, String, Option<String>)> = Vec::new();
   for job in &config.jobs {
       let existing = get_job_by_name(pool, &job.name).await?;
       let existing_resolved = existing
           .as_ref()
           .filter(|db_job| db_job.config_hash == compute_config_hash(job) && db_job.enabled)
           .map(|db_job| db_job.resolved_schedule.clone());
       batch_input.push((job.name.clone(), job.schedule.clone(), existing_resolved));
   }
   let resolved_map: std::collections::HashMap<String, String> =
       random::resolve_random_schedules_batch(&batch_input, random_min_gap, &mut rng)
           .into_iter()
           .collect();
   ```
