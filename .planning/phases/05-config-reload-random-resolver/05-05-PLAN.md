**tests/reload_sighup.rs -- SIGHUP reload end-to-end test:**

This test verifies Phase 5 Success Criteria #1 and #5a:

```rust
//! Integration test: config edit -> do_reload -> verify DB state.
//!
//! Tests RELOAD-01, RELOAD-05, RELOAD-07.
//! Does NOT send an actual SIGHUP (that's platform-dependent and flaky in CI).
//! Instead, calls do_reload() directly -- the same code path SIGHUP uses.

use std::collections::HashMap;
use std::io::Write;
use std::path::PathBuf;
use std::time::Duration;
use tempfile::NamedTempFile;

// Steps:
// 1. Create a temp config file with 2 jobs (job-a, job-b).
// 2. Create an in-memory SQLite DB, run migrations.
// 3. Call sync_config_to_db() for initial load.
// 4. Verify both jobs exist in DB (get_enabled_jobs).
// 5. Edit the config file: remove job-b, add job-c, change job-a's schedule.
// 6. Call do_reload() with the updated config path.
// 7. Assert:
//    - ReloadResult.status == Ok
//    - ReloadResult.added == 1 (job-c)
//    - ReloadResult.updated == 1 (job-a changed)
//    - ReloadResult.disabled == 1 (job-b removed)
//    - get_job_by_name("job-b") returns enabled=false
//    - get_job_by_name("job-c") returns enabled=true
//    - get_job_by_name("job-a").config_hash changed
```

The test must:
- Use `cronduit::db::DbPool::new_sqlite_memory()` or the equivalent test helper (check existing patterns in `tests/` directory).
- Write TOML config to a `tempfile::NamedTempFile`.
- Call `sync_config_to_db` for initial sync, then `do_reload` for the edit.
- Assert all counts and DB states.

**tests/reload_inflight.rs -- In-flight run survival test:**

This test verifies Phase 5 Success Criteria #5b:

```rust
//! Integration test: reload does not cancel in-flight runs.
//!
//! Tests RELOAD-06.

// Steps:
// 1. Create a temp config with 1 job (job-a, type=command, command="sleep 5").
// 2. Create DB, initial sync.
// 3. Start a run for job-a (insert a job_runs row with status='running').
// 4. Call do_reload() (even with same config -- just verifying it doesn't touch running state).
// 5. Assert:
//    - The job_runs row for job-a still has status='running' (not cancelled/error).
//    - ReloadResult.status == Ok.
//    - The running row's id is unchanged.
```

Note: This is a DB-level test. We don't actually spawn a process -- we insert a `status='running'` row and verify reload doesn't modify it. The actual JoinSet isolation is an implementation property (spawn tasks hold cloned data), but the DB state is what we can verify.

```rust
//! Integration test: @random resolved_schedule stability across reloads.

// 7. Call do_reload() with modified config.
// 8. Read job-a again -> assert resolved_schedule has hour=15 (re-randomized).
// 9. Assert the minute field MAY differ from step 3 (re-randomized).
```
