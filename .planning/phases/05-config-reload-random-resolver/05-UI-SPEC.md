#### 9. Toast JS -- Variable Dismiss Duration

The existing toast JavaScript in `base.html` (line 73, currently hardcoded `3000`) must support variable auto-dismiss:

- `level: "info"` or `level: "success"` -- auto-dismiss after duration from event detail (default 5000ms for reload, 3000ms for Run Now)
- `level: "error"` -- no auto-dismiss; add click-to-dismiss handler

The `showToast` event detail gains an optional `duration` field (milliseconds). If `duration === 0`, the toast persists.

Source: D-01 (5s auto-dismiss), D-02 (persist error toasts)

## Interaction Contracts

### Reload via Settings Page Button

```
1. Operator clicks "Reload Config" button
2. HTMX sends POST /api/reload with CSRF token
3. Server parses config, resolves @random, syncs to DB
4. On success: HX-Trigger header fires showToast with diff summary, level "success", duration 5000
5. On failure: HX-Trigger header fires showToast with error summary, level "error", duration 0
6. Settings page polls or uses hx-trigger="load" to refresh reload timestamp card
```

### Reload via SIGHUP / File Watcher

```
1. Config file edited (file watcher) or SIGHUP received
2. Reload executes server-side; result logged to structured log
3. No immediate UI feedback (no browser connection to push to)
4. Settings page shows updated last_reload timestamp on next page load or HTMX poll
5. Dashboard table reflects new/updated/disabled jobs on next 3s poll cycle
```

### Re-roll via Job Detail

```
1. Operator clicks "Re-roll" button on job detail page
2. HTMX sends POST /api/jobs/{id}/reroll with CSRF token
3. Server re-resolves @random fields for this job, updates resolved_schedule in DB
4. On success: HX-Trigger fires showToast "Schedule re-rolled for {job_name}", level "success", duration 5000
5. Job detail page refreshes resolved schedule display (hx-target on the config card or full page reload via HX-Refresh)
6. On failure (non-random job): HX-Trigger fires showToast "This job has no @random schedule", level "error", duration 0
```

### `@random` Badge Visibility

```
- Dashboard: badge appears inline after resolved_schedule text when job.schedule contains "@random"
- Job Detail: badge appears inline after raw schedule code element
- Badge is read-only (no click interaction)
```

## State Matrix

| State | Visual Treatment | Where Shown |
|-------|-----------------|-------------|
| Reload never performed | "Never" in muted text | Settings reload card |
| Reload success (latest) | Timestamp + diff summary in primary text | Settings reload card |
| Reload failed (latest) | Timestamp in error color + error summary | Settings reload card |
| File watcher enabled | "WATCHING" success badge | Settings watcher card |
| File watcher disabled | "DISABLED" warning badge | Settings watcher card |
| Job has `@random` schedule | `@random` badge pill + resolved schedule display | Dashboard table, job detail |
| Job has fixed schedule | No badge, no resolved schedule section | Dashboard table, job detail |
| Re-roll succeeded | Success toast + updated resolved schedule | Job detail page |
| Re-roll failed (non-random job) | Error toast | Job detail page |
| Reload in progress | Button shows no loading state in v1 (reload is sub-second) | Settings page |

## Accessibility Notes

- Toast messages use `role="status"` and `aria-live="polite"` on the toast container for screen reader announcements
- Error toasts that persist must have a visible dismiss button with `aria-label="Dismiss notification"`
- `@random` badge uses sufficient color contrast (green on green-dim background meets WCAG AA for the font sizes used)
- Re-roll button is a real `<button>` inside a `<form>`, keyboard-accessible by default
- Reload Config button follows the same accessible pattern as the existing Run Now button
