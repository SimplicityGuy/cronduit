<objective>
Create the Docker executor core module that manages the full container lifecycle: create container with labels/volumes/env/network, start it, concurrently stream logs and wait for exit, handle timeout and cancellation via stop_container with 10s grace, drain logs to EOF after exit, extract image digest, and explicitly remove the container after all state is captured.

Purpose: This is the foundation for all Docker job execution. The lifecycle state machine (Creating -> Starting -> Running -> Exited -> LogsDrained -> Removed) must be correct to avoid the auto_remove race (moby#8441).

Output: `src/scheduler/docker.rs` (executor), `src/scheduler/docker_log.rs` (log streaming), bollard added to Cargo.toml.
</objective>

   ```rust
   pub async fn stream_docker_logs(
       docker: Docker,
       container_id: String,
       sender: LogSender,
   )
   ```
   Implementation:
   - Use `bollard::container::LogsOptions` with `follow: Some(true)`, `stdout: Some(true)`, `stderr: Some(true)`, `timestamps: Some(false)`.
   - Call `docker.logs(&container_id, Some(options))` to get the stream.
   - Use `futures_util::StreamExt::next()` in a while-let loop.
   - Match on `LogOutput::StdOut { message }` and `LogOutput::StdErr { message }`:
     - Convert `message` (Bytes) to string via `String::from_utf8_lossy(&message)`
     - Split on newlines: `for line in text.lines()` (handles chunk boundaries that may contain multiple lines)
     - Send each line via `sender.send(make_log_line("stdout"|"stderr", line.to_string()))`
   - On `LogOutput::Console { message }`, treat as stdout.
   - On `Err(e)`, send a system log line `[docker log error: {e}]` and break.
   - Do NOT call `sender.close()` -- the caller manages channel lifecycle.

2. Register module in `src/scheduler/mod.rs`: add `pub mod docker_log;`

3. Complete the `execute_docker` function in `src/scheduler/docker.rs`. After container start + digest extraction, implement the concurrent lifecycle:

   a. Spawn the log streaming task:
      ```rust
      let log_docker = docker.clone();
      let log_container = container_id.clone();
      let log_sender = sender.clone();
      let log_handle = tokio::spawn(
          super::docker_log::stream_docker_logs(log_docker, log_container, log_sender)
      );
      ```

   b. Use `tokio::select!` with three branches:
      - `wait_container` stream via `docker.wait_container::<String>(&container_id, None).next()`:
        - Extract `status_code` from `ContainerWaitResponse` (it is an i64, cast to i32 for ExecResult)
        - After wait resolves: await `log_handle` (D-05: drain logs to EOF)
        - Close sender
        - Determine RunStatus: code 0 = Success, else Failed
      - `tokio::time::sleep(timeout)`:
        - Call `docker.stop_container(&container_id, Some(StopContainerOptions { t: 10 })).await.ok()` (D-04: 10s grace SIGTERM)
        - Await `log_handle`
        - Close sender
        - Set status Timeout with error_message
      - `cancel.cancelled()`:
        - Call `docker.stop_container(&container_id, Some(StopContainerOptions { t: 10 })).await.ok()` (D-06: stop on shutdown)
        - Await `log_handle`
        - Close sender
        - Set status Shutdown with error_message

   c. After the select block (regardless of outcome), explicitly remove the container (DOCKER-06):
      ```rust
      if let Err(e) = docker.remove_container(
          &container_id,
          Some(RemoveContainerOptions { force: true, ..Default::default() })
      ).await {
          tracing::warn!(
              target: "cronduit.docker",
              container_id = %container_id,
              error = %e,
              "failed to remove container"
          );
      }
      ```
      Note: Use `force: true` to handle edge cases where container is still stopping.

   d. Return `DockerExecResult { exec: ExecResult { exit_code, status, error_message }, container_id: Some(image_digest) }` where `image_digest` is the digest extracted via `inspect_container` after start (DOCKER-09).

4. Handle errors at each stage (create, start, inspect, wait) by:
   - Closing the sender
   - Attempting to remove the container if it was created
   - Returning ExecResult with RunStatus::Error and a descriptive error_message

5. Add `use futures_util::StreamExt;` at the top of docker.rs.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cronduit -> Docker socket | Root-equivalent access; any container can be created/started/stopped |
| Config file -> Container spec | Operator-authored config defines what images/volumes/networks are used |
| Docker log stream -> DB | Untrusted container output flows into job_logs table |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Elevation of Privilege | Docker socket access | accept | Socket access is inherent to Cronduit's design; documented in THREAT_MODEL.md; default loopback bind mitigates remote exploitation |
| T-04-02 | Tampering | Volume mount paths | accept | Config is operator-authored and mounted read-only; no user input for volume paths |
| T-04-03 | Information Disclosure | Container labels | mitigate | Labels contain only run_id (integer) and job_name (string), never secrets or env var values |
| T-04-04 | Information Disclosure | Env vars in container | accept | Env vars come from operator config (SecretString in config, plain strings in container env); container inspect not exposed to UI |
| T-04-05 | Denial of Service | Container resource exhaustion | accept | No resource limits in v1; operator responsibility; documented limitation |
| T-04-06 | Information Disclosure | Docker log content | mitigate | Log content is HTML-escaped by existing UI pipeline (Phase 3 XSS hardening); stored as-is in DB |
</threat_model>
