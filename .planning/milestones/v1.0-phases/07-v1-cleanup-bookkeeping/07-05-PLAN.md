# Plan 07-05: Job Detail Run History Auto-Refresh

## Context

**Filed during 06-UAT Test 4 on 2026-04-13.** The user clicked Run Now on a Job Detail
page ~15 times in rapid succession. The new run rows appeared correctly in the Run
History card (HX-Refresh from the earlier run_now fix works), but once the runs
completed, the Run History rows stayed frozen at `RUNNING` — the page never refreshed,
and the user had to manually reload to see them transition to `SUCCESS`.

Screenshot in the UAT thread shows ~15 `RUNNING` rows stacked on top of three
completed `SUCCESS` rows, several seconds after the underlying runs finished.

This is the **same class of bug** as the reload card (Phase 5 UAT) and the Run Now
button (Phase 6 UAT): stale HTML after a background state change. Fixing it on Job
Detail closes the loop so v1.0 ships with a coherent "user never needs to reload"
story across the three interactive pages.

**Scope guardrail:** this is *Job Detail Run History refresh only*. Dashboard-wide
live refresh, SSE for run lists, and "show new run inline" animations are out of
scope and should be deferred.

## Approach

1. Add a partial endpoint `GET /partials/jobs/:job_id/runs` that renders the Run
   History tbody for the given job (reuse the existing query path from `job_detail`).
2. In `templates/pages/job_detail.html`, wrap the Run History tbody in an element
   with `hx-get="/partials/jobs/{{ job.id }}/runs"`, `hx-trigger="every 2s"`,
   `hx-swap="innerHTML"`.
3. Stop the polling when no row has status `running` — either by returning an
   `HX-Trigger` response header that flips the poll off, or by rendering the
   wrapper with a conditional `hx-trigger` based on `any_running` from the handler.
   Prefer the simpler approach.
4. Regression test at HTTP layer (mirror `tests/reload_api.rs` pattern from Plan 07-04):
   assert the new partial endpoint returns 200 + expected HTML and contains the
   expected status badges.

## Out of Scope

- Dashboard (index) page auto-refresh
- Live log streaming on Job Detail (that's SSE on Run Detail only)
- Per-run status push via SSE
- Any UI changes to the Run History card beyond the polling wrapper

## Verification

- Manual: trigger 10+ runs in rapid succession on Job Detail, confirm rows flip
  from RUNNING to terminal state within ~2s of actual completion, confirm polling
  stops once the list is idle (network tab shows no more `/partials/jobs/.../runs`
  requests).
- Automated: new `tests/job_detail_partial.rs` exercises the partial endpoint
  and asserts the response body matches the expected Run History HTML structure.
