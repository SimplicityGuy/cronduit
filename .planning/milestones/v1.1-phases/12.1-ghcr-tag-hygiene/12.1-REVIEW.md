# Phase 12.1: Code Review Report

**Reviewed:** 2026-04-20
**Depth:** standard
**Files Reviewed:** 4
**Status:** issues_found

## Summary

Phase 12.1 removes main-push `:latest`/`:sha-*` overwrites from `ci.yml`, introduces
`main-build.yml` as the sole publisher of the `:main` floating tag, documents a
six-tag contract in `README.md`, and adds `scripts/verify-latest-retag.sh` as a
maintainer verifier for the one-shot OPS-09 `:latest` retag.

Overall quality is high. The GHA workflows follow the project's "never interpolate
`${{ }}` directly into `run:` commands" rule (both newly-added `run:` steps in
`main-build.yml` route variables through `env:` blocks), per-job permissions are
correctly scoped (`packages: write` only on the publish job), top-level perms are
read-only, the workflow is properly gated on `branches: [main]`, concurrency is
correctly configured to self-cancel, and there is a defense-in-depth post-push
assertion that `:main` is actually multi-arch.

The bash verifier script uses `set -euo pipefail`, checks prerequisites with clear
error messages, and isolates its temp dir under a `trap`.

No critical security or correctness issues were found. Two warnings concern hardening
the `main-build.yml` against third-party action supply-chain risk (major-tag pinning
vs. SHA pinning) and one small robustness gap in the bash script (`--help` output
drift risk because it `sed`s line ranges out of the script itself). The remaining
findings are informational style/consistency items.

## Warnings

### WR-01: `main-build.yml` pins third-party Docker actions to major tags, not commit SHAs

**File:** `.github/workflows/main-build.yml:66,69,77,111`
**Issue:** The new `main-build.yml` uses floating major tags for every third-party
action:

- `docker/setup-buildx-action@v3` (L66)
- `docker/login-action@v3` (L69)
- `docker/metadata-action@v5` (L77)
- `docker/build-push-action@v6` (L111)

These actions all have `packages: write` scope in this job and `GITHUB_TOKEN`
exposure via `docker/login-action`. A compromised upstream release on any of the
`vN` moving tags would let an attacker push arbitrary images to
`ghcr.io/simplicityguy/cronduit:main`.

The sibling `release.yml` in this repo has exactly the same behavior for Docker
actions (L38, L74, L77, L85, L167) -- it only SHA-pins the non-docker actions
(`orhun/git-cliff-action` at L65, `softprops/action-gh-release` at L179). So this
is a pre-existing repo-wide posture choice rather than a Phase 12.1 regression.

However, Phase 12.1's stated goal is "GHCR tag hygiene" -- locking down who can
write to GHCR -- so the gap is thematically in scope for this phase even if it
predates it. Worth an explicit decision in this review rather than silent
inheritance.

**Fix:** Either:

1. Accept the status quo and add a top-of-file comment in `main-build.yml`
   documenting the conscious decision to pin to major tags (for dependabot-
   friendly updates) and reference the project's threat model / upstream trust
   assumptions. This aligns with the existing `release.yml` posture.

2. SHA-pin the Docker actions across both `main-build.yml` and `release.yml` for
   consistency, matching the treatment already given to
   `orhun/git-cliff-action@c93ef52f3d0ddcdcc9bd5447d98d458a11cd4f72` and
   `softprops/action-gh-release@b4309332981a82ec1c5618f44dd2e27cc8bfbfda`:

   ```yaml
   - uses: docker/setup-buildx-action@<SHA> # v3.x.x
   - uses: docker/login-action@<SHA> # v3.x.x
   - uses: docker/metadata-action@<SHA> # v5.x.x
   - uses: docker/build-push-action@<SHA> # v6.x.x
   ```

Option 1 is the cheapest and consistent with the repo's current practice; option
2 is the stronger security posture and worth a follow-up phase if the maintainer
decides it's warranted.

### WR-02: `verify-latest-retag.sh --help` uses hard-coded line ranges that drift silently

**File:** `scripts/verify-latest-retag.sh:34-37`
**Issue:** The `--help` handler `sed -n '3,24p' "$0"` prints the literal lines
3-24 of the script. That range covers the usage block today, but any edit that
adds or removes lines above line 24 (adding a new flag, a new prerequisite,
extending the "Why per-platform" paragraph) will silently truncate or include
unrelated lines in `--help` output.

This is a correctness-adjacent quality bug: `--help` is the user-facing contract,
and the contract is anchored to line numbers rather than to content markers.

**Fix:** Anchor the range to content markers rather than line numbers, or extract
the usage block into a function:

```bash
usage() {
  cat <<'EOF'
verify-latest-retag.sh — Assert :latest per-platform digests match :<stable>.

Usage:
  ./scripts/verify-latest-retag.sh              # default :stable-tag = 1.0.1
  ./scripts/verify-latest-retag.sh 1.1.0        # after v1.1.0 ships
  ./scripts/verify-latest-retag.sh --help       # show this block

Prerequisites:
  - docker with buildx plugin
  - jq
  - Read access to ghcr.io/simplicityguy/cronduit
EOF
}

if [ "${1:-}" = "--help" ] || [ "${1:-}" = "-h" ]; then
  usage
  exit 0
fi
```

Alternative (lighter-touch): have `sed` match between sentinel comments instead
of line numbers, e.g. `sed -n '/^# BEGIN_USAGE/,/^# END_USAGE/p' "$0"` with
matching `# BEGIN_USAGE` / `# END_USAGE` comments wrapping the usage block.

## Info

### IN-01: `STABLE_TAG` default `1.0.1` will become stale as new stables ship

**File:** `scripts/verify-latest-retag.sh:40`
**Issue:** `STABLE_TAG="${1:-1.0.1}"` hard-codes the current stable as the
default. After `v1.1.0` ships, a maintainer running the script without an
argument will verify against the wrong source and see spurious divergence.

This is by-design today (per the usage comment at L21: `default :stable-tag =
1.0.1`), and the project plans to update the default in a follow-up; flagging it
so the default bump is explicitly tracked rather than forgotten.

**Fix:** Either (a) drop the default and require the argument (`STABLE_TAG="${1:?tag required — e.g. 1.0.1}"`),
or (b) add a TODO comment at L40 linking to the next-stable-release checklist so
the default bump is not lost. Option (a) is safer because it fails loudly on
misuse.

### IN-02: README "Pinning a specific image tag" sentence has inconsistent tag ordering

**File:** `README.md:67`
**Issue:** The prose lists tags as `:X.Y`, `:latest`, `:rc`, `:main`, but the
table directly below (L84-91) orders them `:X.Y.Z`, `:X.Y`, `:X`, `:latest`,
`:rc`, `:main` (most-specific to least-specific). The prose ordering is a subset
and the omission of `:X.Y.Z` / `:X` is not explained.

Minor quality issue; readers who skim the prose before reading the table may
wonder whether `:X` and `:X.Y.Z` exist.

**Fix:** Align prose ordering with table, e.g.:

```markdown
Pinning a specific image tag in production? See [Docker image tags](#docker-image-tags)
for which tag matches different operator needs (`:X.Y.Z`, `:X.Y`, `:X`, `:latest`,
`:rc`, `:main`).
```

### IN-03: `main-build.yml` lacks explicit `GITHUB_TOKEN` minimization

**File:** `.github/workflows/main-build.yml:73`
**Issue:** `docker/login-action` uses `${{ secrets.GITHUB_TOKEN }}` which inherits
the job-level `packages: write` + `contents: read` scope from L46-48. That's
already minimized compared to the workflow's default, but there's no explicit
comment documenting that the `GITHUB_TOKEN` here is intentionally scoped only to
package push. `release.yml` has the same shape (L81) without a comment.

Not a bug -- per-job `permissions:` is correctly the narrowest scope. Just a
documentation gap that makes future audits slightly harder.

**Fix:** Add a one-line comment above L73:

```yaml
  # GITHUB_TOKEN here carries only job-scoped contents:read + packages:write
  # (see permissions: above L46-48).
  password: ${{ secrets.GITHUB_TOKEN }}
```

### IN-04: `verify-latest-retag.sh` `::error::` prefix has a typo

**File:** `scripts/verify-latest-retag.sh:81`
**Issue:** Line 81 reads `echo "::error:::latest per-platform digests do NOT match :$STABLE_TAG" >&2`.
The GitHub Actions workflow command prefix is `::error::` (two colons, `error`,
two colons). The line as written has `::error::` followed immediately by a
literal `:latest`, which looks correct but is easily misread as a triple-colon
typo. Since this script is executed both by CI (where `::error::` is parsed) and
by maintainers on their local machine (where it is just text), the output is
slightly harder to read out of context.

Compare to `ci.yml:173` which uses the same prefix with a space after: `echo
"ERROR: expected exactly 1 'image: cronduit:ci' line, found ${count}"`.

**Fix:** Add a space after the GHA annotation prefix for human readability:

```bash
echo "::error:: :latest per-platform digests do NOT match :$STABLE_TAG" >&2
```

GitHub Actions tolerates the leading space (it trims). This matches common
convention in other workflows that emit `::error::` annotations.
