  # ${{ github.repository }} preserves the original case of the GitHub owner

**Post-push multi-arch assertion step** — NO DIRECT ANALOG in any existing workflow (novel to this phase). RESEARCH.md § "Code Examples / Example 3" (L680-693) provides the draft:

```yaml
      - name: Assert :main is multi-arch (amd64 + arm64)
        env:
          IMG: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:main
        run: |
          set -euo pipefail
          archs=$(docker buildx imagetools inspect "$IMG" --raw \
            | jq -r '[.manifests[] | select(.platform.os == "linux") | .platform.architecture] | sort | join(",")')
          echo "observed architectures: $archs"
          if [ "$archs" != "amd64,arm64" ]; then
            echo "::error::expected :main to publish amd64+arm64, got $archs"
            exit 1
          fi
```

Routing the image tag through `env:` (not `${{ ... }}` inline) maintains the project-wide convention (release.yml L42-48 comment block, ci.yml step patterns).

#
# This script provides a safe, one-command way to update:
#   - Cargo dependencies (Cargo.lock; with --major, also Cargo.toml via cargo-edit)

**Core verification pattern** — NOVEL. Source is RESEARCH.md § "Code Examples / Example 1" (L607-637). Copy into the script with `STABLE_TAG` as a positional arg default `1.0.1`:

```bash
REPO=ghcr.io/simplicityguy/cronduit
STABLE_TAG="${1:-1.0.1}"   # default; update when v1.1.0 stable ships

# 1. Capture the expected per-platform digests from the stable tag.
echo "=== expected per-platform digests ($REPO:$STABLE_TAG) ==="
docker buildx imagetools inspect "$REPO:$STABLE_TAG" --raw \
  | jq -r '.manifests[] | select(.platform.os == "linux") | "\(.platform.architecture)\t\(.digest)"' \
  | tee /tmp/expected.txt

# 2. Capture :latest digests.
echo "=== observed per-platform digests ($REPO:latest) ==="
docker buildx imagetools inspect "$REPO:latest" --raw \
  | jq -r '.manifests[] | select(.platform.os == "linux") | "\(.platform.architecture)\t\(.digest)"' \
  | tee /tmp/observed.txt

# 3. Diff.
if ! diff -u /tmp/expected.txt /tmp/observed.txt; then
  echo "::error:::latest per-platform digests do not match :$STABLE_TAG" >&2
  exit 1
fi

echo ":latest now mirrors :$STABLE_TAG per-platform digests — OPS-09 satisfied."
```

**Executable bit:** script must be `chmod +x` (matches `scripts/update-project.sh` convention — check with `ls -la scripts/` during PR review).

**Why novel:** Nothing else in the repo parses a Docker manifest list with `jq`. The closest precedent is `ci.yml` L83-88 which does `docker pull | docker tag` for testcontainers mirroring — that's a different operation. The `docker buildx imagetools inspect --raw | jq` chain should be introduced with inline comments explaining OCI manifest list structure so future maintainers can modify it safely.

### Pattern: GHA action version pinning

**Source:** all four existing workflows.

**Observation (mixed strategy — document explicitly):**
- **Major-version pin (`@v4`, `@v5`, `@v6`, `@v3`, `@v2`):** `actions/checkout@v4`, `docker/setup-buildx-action@v3`, `docker/login-action@v3`, `docker/metadata-action@v5`, `docker/build-push-action@v6`, `docker/setup-qemu-action@v3`, `extractions/setup-just@v2`, `taiki-e/install-action@v2`, `Swatinem/rust-cache@v2`.

No third-party actions used, so no SHA pinning required. Matches RESEARCH.md § "Security Domain" V10 row ("All actions are version-pinned (`@v4`, `@v5`, `@v6`) per existing project convention").

## Project Conventions Observed

**Workflow file naming:** kebab-case, descriptive of purpose:
- `ci.yml` (generic PR validation)
- `release.yml` (tag-triggered release)
- `compose-smoke.yml` (specific test: compose smoke)
- `cleanup-images.yml` (specific maintenance: image pruning)

→ `main-build.yml` fits this convention (kebab-case, describes purpose).

**Action pinning style (mixed by trust level):**
- Trusted publishers (`actions/*`, `docker/*`, `dtolnay/*`, `extractions/*`, `taiki-e/*`, `Swatinem/*`): major-version pin (`@v4`, `@v5`, `@v6`, `@stable`).
- Less-trusted third-party publishers (`orhun/*`, `softprops/*`, `dataaxiom/*`): full SHA pin with `# vX.Y.Z` trailing comment.

→ `main-build.yml` uses only `actions/*` + `docker/*` — major-version pin is idiomatic.

**Permission block conventions:**
- Workflow-level `contents: read` default (principle of least privilege) in `ci.yml` and `compose-smoke.yml`.
- `release.yml` uses workflow-level `contents: write + packages: write` because it's a single-job workflow where every step legitimately needs both.
- `ci.yml` uses per-job `packages: write` on the `image` job only (T-01-13 cited in L17 comment).

→ `main-build.yml` can follow either model; RESEARCH.md recommends per-job scoping. Precedent in `cleanup-images.yml` L23-33 which has `contents: read` top-level + `packages: write` per-job is the cleanest match.

**GHCR reference pattern:**
- Workflows: ALWAYS derive from `${{ github.repository }}` via the `tr` lowercase step. NEVER hardcode `ghcr.io/simplicityguy/cronduit`. Rationale: release.yml L22-30 comment documents the v1.0.0 mixed-case bug this prevents.
- Scripts + docs: hardcode `ghcr.io/simplicityguy/cronduit` — they are context-specific.

**Mermaid diagram placement (per CLAUDE.md + project memory):**
- All diagrams MUST be mermaid — CLAUDE.md lists this as a locked constraint.
- README `## Architecture` (L80-105) is the canonical example — fenced ` ```mermaid ` block, flowchart syntax, `classDef` palette at bottom, `class` assignments name-listed (not inline).
