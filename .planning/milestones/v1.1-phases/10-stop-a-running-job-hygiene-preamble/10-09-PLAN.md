<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Template interpolation | askama auto-escapes `{{ run.id }}`, `{{ job.name }}`, `{{ csrf_token }}` for HTML/attribute contexts. Hidden csrf_token input is form-data-encoded by the browser before POST. |
| CSRF cookie vs form input | The form includes a hidden `csrf_token` input whose value is the same token stored in the csrf cookie — handler validates equality in plan 10-07. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-09-01 | Spoofing (CSRF) | Stop form POST | mitigate | Hidden csrf_token input + cookie double-submit validated in plan 10-07's handler. Tests in plan 10-07 lock the 403 response on mismatch. |
| T-10-09-02 | Cross-Site Scripting | Template interpolation of run_id / job_name | mitigate | askama auto-escapes all `{{ }}` substitutions for HTML text and attribute contexts. `aria-label="Stop run #{{ run.id }}"` is attribute context; run.id is i64 so no escape surface anyway. |
| T-10-09-03 | Clickjacking | Malicious iframe wraps the Stop button | accept | v1 trusted-LAN / reverse-proxy posture. Operators are expected to either keep cronduit on loopback or front it with a proxy that sets X-Frame-Options. Phase 10 does not ship clickjacking protection (not v1.1 scope). |

ASVS L1 coverage: V5 Input Validation (browser form encoding of csrf_token), V6 CSRF double-submit (handler-side).
